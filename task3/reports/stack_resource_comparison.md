# 有栈执行流与无栈执行流的栈资源占用对比

## 1. 对比目标

本实验用于验证不同执行流模型下，栈资源是如何被占用和扩展的。重点比较：

- OS 线程
- 有栈绿色线程
- 无栈协程
- Rust Future 状态机
- Tokio task 对 Future 的调度封装

关注指标包括：

- 用户态栈占用
- 内核态栈占用或内核线程栈槽位数量
- Future 状态机大小
- 进程整体内存占用
- 高并发任务数量增加时的资源增长趋势

## 2. OS 线程的栈资源

OS 线程通常涉及两类栈：

- 用户栈：线程在用户态执行普通函数调用、局部变量保存、返回地址保存时使用。
- 内核栈：线程进入内核态时使用，例如系统调用、中断、异常、调度等。

用户态和内核态权限级别不同，不能直接共用同一个栈。因此，一个 OS 线程通常既有用户态线程栈，也有对应的内核线程栈。

在用户态运行时，线程主要使用用户栈；当它通过系统调用进入内核，或者被中断、异常打断时，CPU 切换到该线程对应的内核栈上执行内核代码。返回用户态后，内核栈不再被当前用户代码使用。

这里说“内核栈空闲”，不是说内核栈没有分配，而是说它在当前线程执行用户态代码时没有被持续使用。它仍然作为该内核线程的内核态执行上下文存在。

因此，OS 线程模型的资源特点是：

```text
任务数 N -> N 个 OS 线程 -> N 份用户栈 + N 份内核线程栈
```

当任务数量从 1k 增长到 10k、50k、100k 时，OS 线程模型会快速遇到以下问题：

- 用户栈预留量线性增长。
- 内核线程数量线性增长。
- 内核线程栈槽位数量线性增长。
- 调度和上下文切换开销增大。
- 系统可能因为线程数量、虚拟内存、PID 限制而创建失败。

## 3. 有栈绿色线程的栈资源

有栈绿色线程是用户态调度的执行流。每个绿色线程有自己的用户态栈，但它们不一定各自对应一个 OS 线程。

本实验中的有栈绿色线程使用 `may` coroutine 表示。它的关系可以概括为：

```text
任务数 N -> N 个绿色线程 -> N 份用户态 coroutine 栈 -> 少量 worker OS 线程
```

它和 OS 线程相比，关键区别在于：

- 每个任务仍然有独立用户态栈。
- 多个绿色线程复用少量 OS worker 线程。
- 内核线程栈数量主要由 worker 数决定，不随任务数线性增长。

因此，有栈绿色线程可以显著降低内核线程栈数量，但不能消除每个任务独立用户态栈带来的内存压力。

如果创建 10000 个绿色线程，并且每个绿色线程配置 64 KiB 用户态栈，那么用户态栈预留量大约为：

```text
10000 * 64 KiB = 625 MiB
```

即使只有少量内核线程，用户态栈预留仍然会随任务数线性增长。

## 4. 无栈协程与 Future 状态机

Rust 的 `async fn` 会被编译成 Future 状态机。Future 在 `.await` 处挂起时，不保存一整段调用栈，而只保存恢复执行所需的状态，例如：

- 当前状态编号
- 尚未完成的子 Future
- 跨 `.await` 仍然需要保留的局部变量

因此，无栈协程的关系可以概括为：

```text
任务数 N -> N 个 Future 状态机 -> 少量 Tokio worker OS 线程
```

它和有栈绿色线程的关键区别是：

- 有栈绿色线程：每个任务有独立用户态栈。
- 无栈协程：每个任务没有独立用户态栈，挂起状态保存在 Future 状态机里。

本实验通过 `std::mem::size_of_val` 估算单个 Future 状态机大小。这个大小通常远小于一个线程栈或 coroutine 栈。

需要注意：Tokio task 不等于 Future 本身。Tokio task 是运行时调度 Future 的封装，除了 Future 状态机外，还会带来调度队列、引用计数、计时器等运行时开销。这些开销会体现在 RSS 中。

## 5. 无栈协程与内核栈“空闲”的准确表述

容易误解的一点是：无栈协程并不是为每个任务创建一个“空的内核栈”。

更准确的说法是：

```text
无栈协程任务本身不对应独立 OS 线程，
因此也不对应独立内核线程栈。
大量挂起 Future 复用少量 runtime worker 的内核栈。
```

当一个 async task 挂起等待 I/O 或 sleep 时：

- 它不占用独立用户态栈。
- 它的状态保存在 Future 状态机中。
- 它不拥有独立内核线程栈。
- 后续 I/O 就绪、定时器唤醒、调度等由 runtime worker 和内核机制配合处理。

因此，无栈协程高并发时看到的现象不是“每个任务都有一个低利用率内核栈”，而是：

```text
任务数量很多，但内核线程栈槽位很少。
```

例如 100000 个 async task 运行在 4 个 Tokio worker 上，可能只看到 5 到 7 个左右的内核线程，包括主线程、采样线程和 worker 线程。此时每 1000 个任务对应的内核线程栈槽位可能远小于 1。

这正是无栈协程在 I/O 密集高并发场景下节省栈资源的关键原因。

## 6. 三种模型的对比总结

| 模型 | 用户态栈 | 内核线程栈 | 任务挂起后的状态保存方式 | 高并发扩展趋势 |
| --- | --- | --- | --- | --- |
| OS 线程 | 每任务一份用户栈 | 每任务一份内核线程栈 | 线程上下文 + 栈 | 栈和线程数随任务数线性增长 |
| 有栈绿色线程 | 每任务一份 coroutine 栈 | 少量 worker 内核栈 | 用户态栈保存上下文 | 内核线程少，但用户栈仍线性增长 |
| 无栈协程 | 无独立任务栈 | 少量 worker 内核栈 | Future 状态机 | 栈槽位基本由 worker 数决定 |

可以用下面的方式理解资源增长：

```text
OS 线程:
任务数 N -> 用户栈 O(N) + 内核栈槽位 O(N)

有栈绿色线程:
任务数 N -> 用户栈 O(N) + 内核栈槽位 O(worker)

无栈协程:
任务数 N -> Future 状态机 O(N) + 内核栈槽位 O(worker)
```

## 7. 如何用实验数据验证

本实验新增了几个用于验证内核栈趋势的指标：

| 指标 | 含义 |
| --- | --- |
| `peak_kernel_threads` | 峰值内核线程数 |
| `estimated_kernel_stack_reserved_bytes` | 内核栈预留估算 |
| `kernel_stack_slots_per_1000_tasks` | 每 1000 个任务对应多少个内核线程栈槽位 |
| `peak_running_kernel_threads` | 峰值运行态线程数 |
| `peak_sleeping_kernel_threads` | 峰值睡眠态线程数 |
| `dominant_wait_channel` | 最主要的内核等待点 |

验证方法：

1. 查看 `user_stack_reserved.png`，比较 OS 线程、有栈绿色线程和 async Future 的用户态栈预留。
2. 查看 `kernel_stack_reserved.png`，比较内核线程栈预留估算。
3. 查看 `kernel_stack_slots_per_1000_tasks.png`，重点观察每 1000 个任务对应多少内核线程栈槽位。
4. 查看 `kernel_thread_states.png`，观察线程大多处于运行态还是睡眠态。
5. 查看 `data/failures.log`，确认 OS 线程在较大任务规模下是否因为系统限制失败。

如果实验结果显示：

- OS 线程在 1k 时 `kernel_stack_slots_per_1000_tasks` 接近 1000；
- OS 线程在 10k、50k、100k 失败；
- green_thread 和 async_future 的 `kernel_stack_slots_per_1000_tasks` 随任务数增加快速下降；
- async_future 的用户栈预留为 0，但 Future 状态机和 RSS 随任务数增长；

则可以得出结论：

```text
无栈协程不是让大量内核栈变空，而是避免为大量任务创建独立内核线程栈；
它通过 Future 状态机保存挂起上下文，并用少量 worker 内核线程完成调度。
```
