# task1 用户态并发爬虫实验报告

## 摘要

本实验实现了三种用户态并发爬虫程序：基于进程的爬虫、基于线程的爬虫和基于协程的爬虫。三个程序读取同一份学校官网 CSV 数据，下载各学校官网首页内容，提取网页纯文本，并以学校中文名称作为文件名保存到本地目录。实验通过多次运行基准测试，统计三种模型的平均吞吐率、延时分布、最大延时、内存开销和保存文本总字节数，从而比较不同并发模型在 I/O 密集型任务中的性能特征。

实验结果表明：进程模型在本次测试中平均吞吐率最高，为 17.84 RPS；协程模型吞吐率与进程模型非常接近，为 17.39 RPS，并且最大延时最低；线程模型的 P50 和 P95 延时表现较好，但整体吞吐率最低。三种模型保存的总字节数均为 305,549，说明本次实验中三种实现完成了相同规模的爬取任务，结果具有可比性。

## 一、实验目的

本实验围绕用户态并发编程模型展开，目标如下：

1. 编写基于进程的爬虫程序，每个爬取任务由独立子进程执行。
2. 编写基于线程的爬虫程序，每个爬取任务由独立线程执行。
3. 编写基于协程的爬虫程序，每个爬取任务由 Tokio 异步任务执行。
4. 从学校官网首页下载纯文本内容，并以学校中文名称为文件名保存到本地。
5. 对比进程、线程和协程三种并发模型在延时分布、吞吐率和内存开销上的差异。

## 二、实验环境与项目结构

实验代码位于 `task1` 目录，核心结构如下：


实验运行在 Ubuntu 环境下，使用 Rust 编写程序，并通过 `cargo run --release` 运行三个爬虫程序。输入数据为 `data/schools.csv`，其中每行包含学校中文名称和官方网站地址。基准测试脚本为 `scripts/run_benchmark.sh`。

## 三、实验实现

### 1. 公共逻辑

三个爬虫程序共用 `src/lib.rs` 中的公共逻辑，主要包括：

- 读取 `schools.csv`，解析学校中文名称和官网 URL。
- 下载学校官网首页内容。
- 将 HTML 内容转换为纯文本。
- 按学校中文名称生成输出文件名。
- 保存纯文本文件。
- 统计成功数、失败数、P50、P95、最大延时、吞吐率、保存字节数和内存占用。
- 输出成功学校列表和失败学校原因。

### 2. 基于进程的爬虫

进程版程序为 `process_crawler.rs`。主进程读取学校列表后，为每个学校创建一个子进程。子进程只负责抓取一个学校官网首页，并将抓取结果通过标准输出返回给主进程。主进程等待所有子进程结束后，统一汇总性能指标。

这种方式的优点是隔离性强，单个子进程失败不会直接破坏主进程状态；缺点是进程创建、上下文切换和独立地址空间会带来额外开销。

### 3. 基于线程的爬虫

线程版程序为 `thread_crawler.rs`。主线程读取学校列表后，为每个学校创建一个线程。每个线程执行阻塞式 HTTP 请求，完成后通过 channel 将结果返回主线程。主线程收集所有线程的结果后进行统计。

这种方式实现相对直观，适合中等规模并发任务。但线程属于内核调度对象，当线程数量增多时，线程栈、上下文切换和阻塞等待都会带来一定开销。

### 4. 基于协程的爬虫

协程版程序为 `coroutine_crawler.rs`。程序使用 Tokio 运行时，为每个学校创建一个异步任务。HTTP 请求使用异步客户端执行，任务在等待网络 I/O 时可以主动让出执行权，由运行时调度其他任务继续运行。

这种方式适合网络爬虫这类 I/O 密集型任务，因为大量时间消耗在等待网络响应上，协程可以用较低的切换成本支持较高并发。

## 四、实验指标

本实验统计以下指标：

- 平均 RPS：平均每秒成功完成的请求数，反映整体吞吐率。
- 平均 P50：成功请求延时的中位数，反映典型请求体验。
- 平均 P95：成功请求延时的 95 分位数，反映尾部延时。
- 平均 MAX：成功请求中的最大延时，反映最慢请求情况。
- 平均 MEM：程序运行时统计到的内存占用，单位为 KiB。
- 总 BYTES：成功保存的纯文本总字节数，反映三种模型输出结果规模是否一致。

需要注意的是，进程模型的内存指标如果只统计主进程，可能低估真实总内存开销。因为子进程执行期间也会占用独立地址空间，而主进程在汇总结果时读到的内存值不能完全代表所有子进程的峰值总和。

## 五、实验结果

多次运行后得到的平均结果如下：

| 模型 | 平均 RPS | 平均 P50 (ms) | 平均 P95 (ms) | 平均 MAX (ms) | 平均 MEM (KiB) | 总 BYTES |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| process | 17.84 | 280.0 | 1643.3 | 2355.7 | 2,968 | 305,549 |
| thread | 13.64 | 259.3 | 1014.3 | 2386.0 | 20,836 | 305,549 |
| coroutine | 17.39 | 269.7 | 1284.0 | 1837.3 | 12,619 | 305,549 |

从结果可以看到，三种模型的总 BYTES 完全一致，均为 305,549。这说明三个程序在本次实验中保存了相同规模的网页纯文本内容，可以在同一任务量下进行性能对比。

## 六、结果分析

### 1. 吞吐率分析

进程模型平均 RPS 为 17.84，是三种模型中最高的。协程模型平均 RPS 为 17.39，与进程模型非常接近，仅略低于进程模型。线程模型平均 RPS 为 13.64，明显低于另外两种模型。

进程模型吞吐率较高，说明在本次学校官网爬取规模下，多个子进程能够充分并行地发起网络请求。由于爬虫任务主要等待网络 I/O，进程间的隔离开销没有完全抵消并行带来的收益。

协程模型的吞吐率也很高，符合 I/O 密集型任务的特点。协程在等待网络响应时不会阻塞整个线程，而是由运行时调度其他任务执行，因此能够用较低的任务切换成本维持较高吞吐。

线程模型吞吐率最低，可能原因是每个线程执行阻塞式请求，线程数量增加后会产生更多内核调度成本。同时，线程模型虽然单个请求延时较低，但整体完成时间受到线程调度、阻塞等待和最慢任务的影响，最终吞吐率不如进程和协程。

### 2. 延时分布分析

从 P50 延时看，线程模型最低，为 259.3 ms；协程模型为 269.7 ms；进程模型为 280.0 ms。三者 P50 差距不大，说明在大部分普通请求上，三种并发模型都能较快完成任务。

从 P95 延时看，线程模型最好，为 1014.3 ms；协程模型为 1284.0 ms；进程模型最高，为 1643.3 ms。进程模型 P95 较高，说明它的尾部请求更容易受到子进程创建、进程调度或部分网站响应较慢的影响。

从最大延时看，协程模型最低，为 1837.3 ms；进程模型为 2355.7 ms；线程模型最高，为 2386.0 ms。协程模型最大延时最低，说明在本次测试中异步任务的尾部稳定性较好，最慢请求没有被明显拉长。

综合来看，线程模型在 P50 和 P95 上表现较好，但最大延时较高；协程模型 P50/P95 居中，但最大延时最低，整体尾部表现更稳定；进程模型吞吐率最高，但 P95 较高，尾部延时波动更明显。

### 3. 内存开销分析

统计结果中，线程模型平均 MEM 为 20,836 KiB，协程模型为 12,619 KiB，进程模型为 2,968 KiB。

表面上看，进程模型内存最低，但这一结果需要谨慎解释。当前内存指标主要统计主进程常驻内存，没有完整统计所有子进程在运行期间的峰值内存总和。因此，进程模型真实内存开销通常应高于表格中的主进程数值。

线程模型内存最高，符合线程模型的特点。每个线程都需要线程栈和内核调度相关资源，并发线程越多，内存开销越明显。

协程模型内存低于线程模型。异步任务比线程更轻量，不需要为每个任务分配独立的操作系统线程栈，因此在大量 I/O 并发场景下通常具有更好的内存效率。

### 4. 正确性分析

三种模型的总 BYTES 完全一致，说明在本次实验数据下，它们保存的纯文本总量相同。结合程序输出的 `success_schools` 和 `failed_schools`，可以进一步确认哪些学校抓取成功、哪些学校失败以及失败原因。

爬取失败通常可能来自以下原因：

- 学校官网访问超时。
- 官网拒绝连接或临时不可用。
- HTTP 状态码异常，例如 403、404 或 5xx。
- DNS 解析、TLS 握手或网络波动导致请求失败。
- 部分网站对自动化访问有限流或反爬策略。

## 七、三种模型对比总结

| 模型 | 优点 | 缺点 | 本次实验表现 |
| --- | --- | --- | --- |
| 进程 | 隔离性强，单个任务失败影响小，能利用多进程并行 | 创建成本高，真实总内存开销高，尾部延时波动较大 | 吞吐率最高，但 P95 延时最高 |
| 线程 | 编程模型直观，阻塞 I/O 写法简单，P50/P95 表现较好 | 线程栈和内核调度成本较高，大量线程时内存开销明显 | P50/P95 最低，但吞吐率最低，内存最高 |
| 协程 | 任务轻量，适合大量网络 I/O，高并发下内存效率较好 | 异步编程复杂度更高，需要运行时支持 | 吞吐率接近进程，最大延时最低，内存低于线程 |

## 八、实验结论

本实验说明，网络爬虫属于典型 I/O 密集型任务，不同用户态并发模型会在吞吐率、延时和内存开销上表现出不同特征。

在本次实验数据中，进程模型的平均吞吐率最高，达到 17.84 RPS，但 P95 延时也最高，说明尾部请求波动较大。线程模型的 P50 和 P95 延时最好，分别为 259.3 ms 和 1014.3 ms，但平均吞吐率最低，且内存占用最高。协程模型的吞吐率为 17.39 RPS，接近进程模型，同时最大延时最低，为 1837.3 ms，内存占用也低于线程模型，整体表现较均衡。

如果任务规模较小并且强调隔离性，可以选择进程模型；如果希望代码保持同步风格、并发规模中等，可以选择线程模型；如果面对大量网络 I/O 请求并追求较高吞吐率和较低内存开销，协程模型更加合适。

因此，针对本实验中的学校官网首页爬取任务，协程模型在吞吐率、尾部延时和内存开销之间取得了较好的平衡，是更适合扩展到大规模网络爬取场景的实现方式。

## 九、实验不足与改进方向

本实验仍存在一些可改进之处：

1. 当前真实网络实验会受到公网质量、学校官网服务器状态、DNS、TLS、CDN 和限流策略影响，结果存在波动。
2. 进程模型的内存统计没有完整覆盖所有子进程的运行期峰值，后续可以统计每个子进程 RSS 并求和。
3. HTML 转纯文本采用轻量规则处理，复杂页面可能需要更完整的 HTML 解析库。
4. 后续可以增加并发数控制，比较不同并发规模下三种模型的性能变化。
5. 后续可以分别记录成功学校数量、失败学校数量和失败原因分布，使正确性分析更细致。

总体而言，本实验完成了三种用户态并发爬虫的实现和性能对比，验证了进程、线程和协程在 I/O 密集型任务中的不同性能特征。
